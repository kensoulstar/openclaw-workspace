# Simplify Skill

全面审查代码的复用性、质量和效率，并修复发现的问题。

## 审查维度

### 1. 复用性审查

检查是否存在重复代码：

| 检查项 | 说明 |
|--------|------|
| **重复函数** | 是否有功能相同的重复函数 |
| **相似逻辑** | 是否有可统一的相似代码模式 |
| **工具函数** | 是否可以利用现有工具函数替代手写逻辑 |
| **共享模块** | 是否可以提取公共模块 |

### 2. 质量审查

检查代码质量：

| 检查项 | 说明 |
|--------|------|
| **冗余状态** | 是否存在重复的状态存储 |
| **参数膨胀** | 函数参数是否过多 |
| **字符串硬编码** | 是否使用魔法字符串 |
| **过度封装** | 是否有不必要的抽象层 |
| **无用代码** | 是否有注释掉的死代码 |
| **不必要的注释** | 是否有关于"做什么"而非"为什么"的注释 |

### 3. 效率审查

检查性能问题：

| 检查项 | 说明 |
|--------|------|
| **重复计算** | 是否有不必要的重复计算 |
| **串行执行** | 是否有机会并行但未并行的操作 |
| **内存泄漏** | 是否有未清理的监听器或缓存 |
| **不必要的检查** | 是否有TOCTOU反模式 |
| **过度读取** | 是否读取了整个文件但只用到部分 |

## 工作流程

### Phase 1: 识别变更

首先了解需要审查的代码：

```
- 查看 git diff（如有版本控制）
- 识别最近修改的文件
- 确定审查范围
```

### Phase 2: 并行审查

启动三个审查维度并行分析：

```
1. 复用性审查 → 找出可复用的代码
2. 质量审查 → 找出质量问题
3. 效率审查 → 找出性能问题
```

### Phase 3: 修复问题

根据审查结果修复问题：

```
优先级：
1. 高优先级：明显bug、安全问题
2. 中优先级：代码质量、可维护性
3. 低优先级：优化建议
```

## 修复原则

### ✅ 应该做的

- 保持功能不变
- 小步修改，逐步验证
- 优先修复明显问题
- 保持代码风格一致

### ❌ 不应该做的

- 不要为了"优化"而破坏功能
- 不要过度工程化
- 不要引入新的复杂度
- 不要删除看似无用但实际有用的注释

## 注意事项

- 简化前先理解代码意图
- 保留必要的业务逻辑注释
- 简化后务必验证功能正常
- 对于复杂逻辑，考虑添加解释性注释
